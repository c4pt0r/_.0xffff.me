Agents don’t just “think”. They **produce artifacts**: plans, logs, cached responses, patches, reports, traces.

Those artifacts are naturally **file-shaped** — easy to diff, inspect, replay, and share.

But once artifacts pile up, “a folder full of files” stops being a system. You need structure: querying, filtering, deduping, joining, ranking, scheduling, debugging.

That’s where **SQL** becomes the management layer.

The sweet spot is not choosing filesystem *or* Postgres. It’s putting them in the same workflow:

- **Filesystem** stays the honest working state (durable artifacts)
- **Postgres** turns state into something computable (governance and retrieval)
- **SQL** becomes the glue — without glue code

## Files as first-class artifacts
Files are the native format of agent work:

- markdown for plans and reports
- JSONL for traces
- diffs for code changes
- cached HTTP responses
- intermediate datasets

They’re durable, human-readable, and naturally versionable.

## SQL as the “compute and governance” layer
SQL answers the operational questions you’ll inevitably ask:

- Which run failed most recently?
- Which chunks belong to which document?
- Which outputs were generated by model X?
- Which caches are stale?
- Which branch performed better?

That’s not “AI magic” — that’s basic system hygiene.

## The killer feature: files participate in SQL workflows
When your runtime can read/write files from SQL, artifacts stop being external blobs and become queryable inputs and outputs.

Write an artifact:

```sql
select fs9_write('/reports/hello.txt', 'hello from db9');
```

Read and inspect it:

```sql
select fs9_read('/reports/hello.txt');
select fs9_exists('/reports/hello.txt'), fs9_size('/reports/hello.txt');
```

Treat files as query sources (CSV/JSONL → relations):

```sql
select * from extensions.fs9('/data/users.csv');

select _line_number, line
from extensions.fs9('/logs/app.jsonl')
where line->>'level' = 'error';
```

## A practical pattern: keep artifacts as files, track them in tables
Let artifacts remain files, but register them in Postgres so you can query them.

```sql
create table artifacts (
  id serial primary key,
  path text not null unique,
  kind text not null,
  metadata jsonb not null default '{}',
  created_at timestamptz not null default now()
);

insert into artifacts (path, kind, metadata)
values (
  '/reports/final-answer.md',
  'report',
  '{"task":"support-bot","model":"gpt-5"}'
);

select a.path, fs9_read(a.path)
from artifacts a
where a.kind = 'report'
order by a.created_at desc
limit 1;
```

Artifacts stay transparent. SQL provides order.

## Punchline
Files are the agent’s working memory. SQL is the management layer.

Put them together and you get agent systems that are not only powerful, but actually **operable**: inspectable, debuggable, reproducible, and easy to evolve.
